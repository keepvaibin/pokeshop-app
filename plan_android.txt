1. PROJECT OVERVIEW
================================================================================

Pokeshop is a small TCG (Trading Card Game) shop that sells Pokémon cards and
accessories, primarily to UCSC (UC Santa Cruz) students. The web frontend is a
Next.js 16 app. This document specifies a from-scratch native Android rewrite.

KEY FACTS:
- Authentication: Google Sign-In (@ucsc.edu emails only) OR email+password with
  an "access code" invitation system for non-UCSC users
- Users can browse products, add to cart, and check out
- Admin users get a completely different view (dashboard, dispatch, inventory
  management, etc.)
- Checkout supports scheduling campus pickup timeslots or ASAP downtown meetups
- Payment is handled in person (Venmo/Zelle/PayPal/Cash at pickup) — NO
  in-app payment processing, NO Stripe or credit card integration
- Trade-in system: users can trade Pokémon cards for store credit to offset
  their order total
- Orders are fulfilled in person at pickup, not shipped
- The app folder lives at: /app/android/ in the pokeshop repo
- Django backend must be reachable from the phone (production HTTPS URL, not
  localhost)

3. BACKEND API CONFIGURATION
================================================================================

The backend is a Django REST Framework API. The base URL is configurable.

BASE URL (production): set this in a BuildConfig field, e.g.:
  - BuildConfig.API_BASE_URL

During development on a physical device, you cannot use localhost. Use either:
  - An ngrok tunnel pointing to port 8000, or
  - The machine's LAN IP (e.g., http://192.168.x.x:8000)
  - Or deploy the Django backend to a cloud host and use HTTPS

All API calls require a JSON Content-Type header unless uploading files
(multipart/form-data for image uploads).

AUTH HEADER: "Authorization: Bearer <access_token>"
  - Access tokens expire after 60 minutes
  - Refresh tokens expire after 7 days
  - On any 401 response, immediately attempt to refresh the token using
    POST /api/token/refresh/ with { "refresh": "<refresh_token>" }
  - If refresh fails, clear stored tokens and redirect to Login screen
  - Implement token refresh as a thread-safe singleton (only one refresh at a
    time; queue concurrent requests and replay them after refresh succeeds)

RATE LIMITS (important, enforce these in the UI):
  - Anonymous users: 120 requests/minute
  - Authenticated users: 600 requests/minute
  - Checkout endpoint: 10 requests/hour — show a cooldown message if hit

CORS: The backend allows all origins in development mode. For production, the
backend will be configured with the app's domain. For native apps, CORS is not
enforced (it's a browser concept), so this is not a concern.

4. AUTHENTICATION SYSTEM
================================================================================

There are three ways to authenticate:

--- 4.1 GOOGLE SIGN-IN (@ucsc.edu only) ---
- Use the Google Identity Services Credential Manager API for Android
- The user selects their Google account; you receive an ID token (JWT)
- POST that token to: POST /api/auth/google/
  Body: { "token": "<google_id_token>" }
  Response: { "access": "...", "refresh": "...", "user": { ...UserData } }
- Store access token + refresh token in EncryptedSharedPreferences
- The backend ONLY accepts @ucsc.edu Google accounts (validated server-side)
  If the email is not @ucsc.edu, the backend returns 400; show:
  "Only @ucsc.edu Google accounts are supported."

--- 4.2 EMAIL/PASSWORD LOGIN ---
- For non-UCSC users who were given an access code
- POST /api/auth/login/
  Body: { "email": "...", "password": "..." }
  Response: { "access": "...", "refresh": "...", "user": { ...UserData } }
- Same token storage as Google Sign-In

--- 4.3 ACCESS CODE REGISTRATION (Two-Step) ---
Step 1 — Validate code:
  POST /api/auth/validate-access-code/
  Body: { "code": "<access_code>" }
  Response: 200 OK if valid, 400 if invalid/expired/maxed out

Step 2 — Create account:
  POST /api/auth/register/
  Body: {
    "email": "...",
    "password": "...",
    "first_name": "...",
    "last_name": "...",
    "access_code": "<code_from_step1>"
  }
  Response: { "access": "...", "refresh": "...", "user": { ...UserData } }

--- 4.4 TOKEN REFRESH ---
POST /api/token/refresh/
Body: { "refresh": "<refresh_token>" }
Response: { "access": "...", "refresh": "..." }  (rotation enabled)

--- 4.5 USER DATA STRUCTURE ---
The /api/auth/user/ endpoint (GET) returns the current user:
{
  "id": 1,
  "email": "user@ucsc.edu",
  "username": "user",
  "is_admin": false,       // ← THIS is the key field for admin detection
  "first_name": "Jane",
  "last_name": "Doe",
  "nickname": "Janie",
  "discord_id": "12345678",
  "discord_handle": "janie#1234",
  "pokemon_icon": {
    "pokedex_number": 25,
    "filename": "025.png",
    "display_name": "Pikachu",
    "region": "Kanto"
  },
  "strike_count": 0,
  "is_restricted": false
}

--- 4.6 ADMIN DETECTION ---
A user is an admin if user.is_admin == true.
When is_admin is true:
- Show Admin section in bottom nav or side drawer
- Home screen shows AdminDashboard instead of the storefront
- Extra menu items appear
- Write endpoints (create/edit/delete inventory, dispatch orders, etc.) become
  available

--- 4.7 STRIKES ---
If user.strike_count > 0, show a warning banner on the Home screen and on the
Cart screen. Strikes can restrict checkout ability (user.is_restricted = true).

--- 4.8 DISCORD LINKING (Optional) ---
- From Settings screen, user can link their Discord account
- GET /api/auth/discord/initiate/ (authenticated) — returns { "auth_url": "..." }
- Open that URL in a Custom Chrome Tab or external browser
- The backend redirects back to the frontend with ?discord=linked|cancelled|error
- For the native app: use a deep link (custom scheme, e.g., pokeshop://discord/callback)
  and configure the backend to redirect to that scheme
- Update the local user data after successful link (refresh user from API)

5. NAVIGATION ARCHITECTURE
================================================================================

Use a single NavHost with bottom navigation (BottomNavigationBar) for the main
app flows. Admin screens use a separate admin navigation section.

--- BOTTOM NAV (Regular User) ---
Tab 1: Home          (icon: house)
Tab 2: Shop          (icon: shopping bag / store)
Tab 3: Cart          (icon: cart with badge for item count)
Tab 4: Orders        (icon: receipt / list)
Tab 5: Profile       (icon: person)

--- BOTTOM NAV (Admin User) ---
Tab 1: Dashboard     (icon: grid / dashboard)
Tab 2: Shop          (same as regular, admin can still browse)
Tab 3: Cart          (icon: cart)
Tab 4: Dispatch      (icon: zap / lightning — for order fulfillment)
Tab 5: Admin Menu    (icon: settings / wrench — opens admin sub-menu)

The admin user can TOGGLE between "Admin View" and "Storefront View" from the
home screen (same as web: a toggle button in the admin dashboard header).

Public (no auth required):
  home
  shop/{categorySlug?}
  new_releases
  search?q={query}
  product/{itemSlug}

  cart

  checkout_success/{orderId}

  settings_pokemon_picker  (full-screen picker modal/sheet)

Admin only:
  admin_dashboard

7. DARK / LIGHT MODE
================================================================================

The web app does not currently implement dark mode. It is a light-only design.

HOWEVER, for the native Android app, implementing dark mode is recommended
because Android users expect it. Define two theme variants:

LIGHT MODE (matches web exactly):
  Background: #F0F0F0
  Surface: #FFFFFF
  Text Primary: #000000
  Text Secondary: #4D4D4D

DARK MODE (suggested values, maintain brand feel):
  Background: #121212
  Surface: #1E1E1E
  Surface Variant: #2C2C2C
  Text Primary: #FFFFFF
  Text Secondary: #CCCCCC
  Primary Blue: #4D9FE0  (slightly lighter for contrast on dark)
  Yellow Accent: #FDB71D  (same)
  Red: #FF4444  (brighter for dark)
  Border: #333333
  Input bg: #2A2A2A

Use MaterialTheme with a DynamicColorScheme for Android 12+ opt-in, but always
fall back to the brand colors above for older versions.

Respect the system setting (AppCompatDelegate.MODE_NIGHT_FOLLOW_SYSTEM) by
default, with a manual override option in Settings → Appearance.

==========================================================
SCREEN: SPLASH / LOADING
==========================================================
PURPOSE: App entry point. Determines auth state and routes user.
LAYOUT:
  - Full-screen blue (#0054A6) background
  - Pokeshop logo centered (or Pokéball icon)
  - Animated loading indicator
LOGIC:
  1. On app start, check EncryptedSharedPreferences for stored tokens
  2. If access token present AND not expired: call GET /api/auth/user/ to
     validate and refresh user data; navigate to Home
  3. If access token expired but refresh token present: call
     POST /api/token/refresh/; on success store new tokens and go to Home;
     on failure clear tokens and go to Login
  4. If no tokens: go to Login
DURATION: No artificial delay; just wait for the token check API call

9. TRADE-IN SYSTEM (Full Specification)
================================================================================

The trade-in system allows customers to offset their order total by bringing
in Pokémon cards for the shop to buy.

--- CREDIT FORMULA ---
base_market_price: fetched from TCG oracle database (TCGCSV/pokemontcg.io)
condition_multiplier:
  Near Mint:        1.00
  Lightly Played:   0.85
  Moderately Played:0.70
  Heavily Played:   0.50
  Damaged:          0.30
credit_percentage: from store settings (e.g., 85%)

card_credit = base_market_price × condition_multiplier × (credit_percentage/100)

total_trade_credit = sum(card_credit for all accepted cards)

--- CARD ENTRY FLOW ---
1. User taps [Add Trade Card] in checkout step 5
2. Opens TCGCardSearchSheet (bottom sheet):
   - Text input, debounced search via GET /api/inventory/tcg-search/?q=...
   - Shows search results: card name, set, rarity, current market price
   - Tap a result: auto-fills card_name, tcg_product_id, base_market_price,
     rarity, estimated_value
   - "Enter manually" toggle: free-text entry (no oracle lookup)
3. Back in trade card list:
   - Show card name, condition dropdown, estimated credit
   - Optional: [Add Photo] button (important for condition verification)
   - Photo: compress to <2MB before attaching
   - "Wanted Card" badge: show if card name matches any active wanted card
     GET /api/inventory/wanted/ and do client-side name matching

--- MODES ---
"All or Nothing" (default): Admin approves or rejects the entire trade
"Allow Partial": Admin can accept/reject individual cards with per-card overrides

The user selects trade_mode in the checkout if the store allows partial.
The current web frontend uses "allow_partial" as a user choice in checkout.

--- TRADE OVERAGE ---
If total trade credit > order subtotal:
  trade_overage = total_credit - subtotal
  Show: "Trade credit exceeds order total. You'll receive $X.XX in store credit"
  The backend stores this overage on the Order model.

--- ADMIN TRADE REVIEW ---
In Dispatch → TRADES tab:
  - Admin sees each trade card with oracle-fetched price vs. user estimate
  - Can accept/reject per card (partial mode) or all-or-nothing
  - Can override the value of individual cards
  - Can send a counteroffer: custom message + expires_at datetime
  - User receives notification (via Discord DM or push notification)
  - User can accept or decline counteroffer from Order Detail screen

10. PURCHASE LIMITS
================================================================================

Per item, there are up to 3 independent limits:
  max_per_user:      daily limit (noon to noon reset)
  max_per_week:      rolling 7-day window
  max_total_per_user:lifetime limit (null = no limit)

At checkout and in the quick view sheet, the app checks:
GET /api/orders/purchase-limits/
Response: { "item_id_1": { "remaining_today": 2, "remaining_week": 5 }, ... }

When adding to cart: client-side enforce qty ≤ remaining
At checkout: server also enforces (so double-check on submit)

If a user has already hit their limit, show:
"You've reached your purchase limit for [Item Name]."

11. COUPON SYSTEM
================================================================================

Coupons are validated in real-time during checkout step 6.
POST /api/orders/validate-coupon/ with { "code": "...", "order_total": X,
  "item_ids": [id1, id2, ...] }

Response fields:
  valid: bool
  discount: decimal (amount to subtract)
  discount_type: "fixed" | "percent"
  message: string (shown to user)
  requires_cash_only: bool (if true, hide non-cash payment options)

If requires_cash_only and user had selected Venmo/Zelle/PayPal:
  Show: "This coupon requires cash payment. Please go back and select Cash."
  Gray out non-cash payment options.

12. TIMESLOT SYSTEM
================================================================================

GET /api/orders/active-timeslots/
Returns two arrays:
  recurring: [{ id, day_of_week, start_time, end_time, location, max_bookings,
               bookings_for_date: { "2026-05-18": 3 } }]
  one_time: [{ id, start, end, max_bookings, current_bookings }]

Display logic for recurring timeslots:
  - day_of_week: 0=Monday, 1=Tuesday, ..., 6=Sunday
  - For each slot, calculate the next N occurrences (next 3 weeks)
  - For each occurrence date, check bookings_for_date to show capacity
  - "X spots left" where X = max_bookings - bookings_for_date[date]
  - Date must be >= today + 2 days (advance purchase window)
  - If is_ooo = true and ooo_until is set: start from ooo_until date + 1 day

When user selects a slot: store { recurring_timeslot_id, pickup_date } or
{ pickup_timeslot_id } for one-time slots.

to support FCM. Plan for this as a future backend task, but build the client-side

14. STATE MANAGEMENT & ARCHITECTURE
================================================================================

Use MVVM + Repository pattern:

UI Layer (Compose screens)
  ↓ observes
ViewModel (holds UI state via StateFlow/UiState sealed class)
  ↓ calls
Use Cases / Repositories
  ↓ calls
Remote Data Source (Retrofit services)
Local Data Source (DataStore, EncryptedSharedPreferences)

Each major feature has:
  - Repository interface + implementation
  - ViewModel with UiState sealed class:
    sealed class UiState<T> {
      object Loading : UiState<Nothing>()
      data class Success<T>(val data: T) : UiState<T>()
      data class Error(val message: String) : UiState<Nothing>()
    }
  - Compose screens collect state as collectAsState()

CART STATE:
  Managed in CartRepository (singleton via Hilt)
  Backed by server (GET/PATCH/DELETE /api/orders/cart/)
  Local cache in memory for the session
  Observable cart item count for badge display

AUTH STATE:
  AuthRepository (singleton) backed by EncryptedSharedPreferences
  Exposes: currentUser: StateFlow<User?>, isLoggedIn, isAdmin
  Token refresh handled transparently by an OkHttp interceptor

15. SECURITY CONSIDERATIONS
================================================================================

1. STORE TOKENS IN EncryptedSharedPreferences (not plain SharedPreferences)
   Do NOT store tokens in regular files or plain preferences.

2. Never log token values, even in debug builds.

3. SSL PINNING: Consider pinning the backend certificate in production.
   Use OkHttp CertificatePinner for the production backend domain.

4. CERTIFICATE VALIDATION: Never use setHostnameVerifier that accepts all hosts.

5. RATE LIMITING: Respect backend rate limits. Implement exponential backoff
   on 429 responses.

6. DEEPLINK HANDLING: pokeshop:// scheme deep links should only be processed
   when they come from the expected sources (Discord OAuth, etc). Validate the
   state parameter in OAuth callbacks.

7. IMAGE UPLOAD: Validate file type client-side before upload. Only JPEG/PNG.
   Compress images to max 2MB before sending.

8. INPUT SANITIZATION: While the backend sanitizes inputs, do not send
   empty/null values where the API expects data.

9. ADMIN ROUTES: Even though the backend enforces admin permissions, hide admin
   UI elements entirely from non-admin users client-side.

16. IMAGE HANDLING
================================================================================

Product images:
  - Loaded from backend URLs via Coil
  - Use CrossFade transitions
  - Placeholder: gray shimmer skeleton
  - Error fallback: Poké Ball icon
  - Cache: Coil's built-in memory + disk cache
  - Format: backend serves JPEG/PNG; Coil handles both

Profile images / Pokémon icons:
  - Served from /api/auth/pokemon-icons/ endpoint
  - Filenames like "025.png" served from static files
  - Small circular images; cache aggressively

Trade card photos (upload):
  - User picks from gallery or takes photo with camera
  - Compress with Android ImageDecoder + Bitmap compression to < 2MB
  - Show upload progress while submitting
  - Sent as multipart/form-data

Banner images:
  - Can be URLs or uploaded files
  - Load with Coil, full-width, aspect ratio maintained

19. WHAT DOES NOT APPLY TO MOBILE (differences from web)
================================================================================

1. NEXT.JS SERVER RENDERING: The web app uses server-side rendering for SEO.
   Android has no concept of SSR. All data is fetched client-side.

2. SERVER-SIDE COOKIES: The web uses an 'auth_hint' cookie to optimize SSR.
   Not needed in the native app.

3. SWR / STALE-WHILE-REVALIDATE: The web uses SWR for data fetching.
   On Android, use Retrofit + coroutines + ViewModel for equivalent patterns.

4. DISCORD DM NOTIFICATIONS: Replace entirely with FCM push notifications.
   Build the client infra now; the backend FCM integration is a separate task.

5. FRAMER MOTION animations: Replace with Compose AnimatedVisibility,
   AnimatedContent, and spring/tween animations.

6. NEXT/IMAGE OPTIMIZATION: Coil handles this natively on Android.

7. RICH TEXT EDITOR (react-quill): For admin inventory creation, use a WebView
   wrapping a minimal quill editor loaded from local HTML assets, OR use a
   Compose markdown editor. Product descriptions are HTML and must be rendered
   as HTML (use a WebView or HtmlText composable to render them in read mode).

8. DRAG-AND-DROP for image reordering: Use ReorderableColumn from a Compose
   DnD library (e.g., sh.calvin.reorderable or similar).

9. CONFETTI on checkout success: Use a Compose particles library or KonfettiKit.

10. PRINT STYLES: Not applicable to Android.

11. ACCESS CODE flow is the same, just in a native UI instead of web forms.

12. GOOGLE HOSTED DOMAIN RESTRICTION (@ucsc.edu): The Android Google Sign-In
    SDK does not filter by hosted domain natively. You MUST still validate on
    the server (the backend checks the hd claim). But you may want to show a
    helpful error message on the client if login fails due to wrong domain.
